Dear Sensei, I have been inspecting the correct use of subspaces in my project based on the TPAMI paper and the Venus data you sent me. My current main experiment is on OSCD (The Sentinel-2 before/after satellite image dataset for binary change detection). For each location, there is a pre-change image and a post-change image, each with 13 Sentinel-2 bands.

At the moment, I generate subspaces from the multi-channel images like this: each valid pixel is treated as one 13-dimensional vector, where the 13 values are the Sentinel-2 band values at that pixel location. So for one location I build:
X_pre  in R^(13 x N)
X_post in R^(13 x N)
where N is the number of valid pixels. For example, in Beirut image, one of the change detection OSCD dataset entries, the image size is 1180 x 1070, so N = 1,262,600. Thus each side gives a 13 x 1,262,600 matrix. With PCA rank 6 this produces one 13 x 6 PCA subspace for the pre-change image and one 13 x 6 PCA subspace for the post-change image. Then DS compares these two subspaces. So my current implementation is not one subspace per band, and not one whole satellite image as one vector. It is one PCA subspace per date image, built from all valid 13-band pixel vectors in that image.

About the Venus data and experiment with KGDS, I reproduced the data and it helped me understand the difference between the TPAMI image-set setting and my satellite-image adaptation:

TPAMI Venus:
  one sample vector = one whole downsampled image view
  vector entries = 3024 pixel values
  number of samples = 300 views of the same sculpture
  subspace = one image-set subspace for that sculpture

My current implementation:
  one sample vector = one valid Sentinel-2 pixel
  vector entries = 13 band values at that pixel location
  number of samples = all valid pixels in one satellite image/date
  subspace = one PCA subspace for the pre-change image and one PCA subspace for the post-change image

I understand that this is an adaptation, not the original DS formulation. In my project I only have one multispectral image per date/location, not many views of the same scene, so I adapted the “set of samples” idea by using the valid pixel vectors inside one image as the sample set. Since OSCD has only two dates per location, it fits DS/KDS more naturally than GDS/KGDS. For GDS/KGDS, I am considering MultiSenGE, a multi-temporal Sentinel-2 dataset I previously tested. There I could build one subspace per date image and extract differences across multiple dates. The main limitation is that it is not labeled like OSCD, so interpretation would require clustering, temporal analysis, or another evaluation strategy.

If you think this sample definition of pixel-wise difference is not appropriate, I would like to discuss whether I should instead use local patches approach, sliding window approach, multiple dates approach for GDS/KGDS, or another subspace construction.
